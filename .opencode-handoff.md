## Estado — Feature: Roles Admin/User + Panel Admin ✅

### Fase 1 — Backend: DB + tipos + seed admin ✅
- Columna `role TEXT NOT NULL DEFAULT 'user'` agregada a tabla `users` via migración
- Seed automático: si existe env var `ADMIN_EMAIL`, se crea/promueve ese usuario a `role='admin'`
- Tipo `User` en `server/src/types.ts` extendido con `role: 'admin' | 'user'`

### Fase 2 — Backend: JWT + middleware + endpoint ✅
- `generateToken()` firma `role` en payload JWT
- `AuthPayload` extendido con `role`
- Middleware `requireAdmin` (403 si role !== 'admin')
- `shapeUser()` en auth.ts y users.ts incluye `role`
- `GET /users` con `authenticateToken` + `requireAdmin` — lista todos los usuarios

### Fase 3 — Frontend: Tipos + auth state ✅
- Tipos `AuthUser` y `User` extendidos con `role?: 'admin' | 'user'`
- `authService.ts`: persiste/recupera `userRole` en localStorage
- `AuthContext`: role fluye automáticamente via `AuthUser`

### Fase 4 — Frontend: AdminPanel + routing ✅
- `src/services/userService.ts` — `fetchAllUsers(token)` llama a `GET /users`
- `src/components/admin/AdminPanel.tsx` — tabla con todos los usuarios (id, email, nombre, rol, registro)
- Ruta `/admin` en Router — renderiza AdminPanel; si no autenticado o no admin, muestra "Acceso denegado"
- Header muestra link "Admin" (púrpura) solo si `user.role === 'admin'`
- 6 tests nuevos en `admin.test.js` (role persistence + fetchAllUsers)

### Fase 5 — Backend tests ✅
- `vitest.config.server.js` — config para tests del servidor (entorno node)
- `server/__tests__/middleware/auth.test.ts` — 4 tests para `requireAdmin` (sin auth, role user, role admin, role vacío)
- `server/__tests__/routes/users.test.ts` — 4 tests para `GET /users` (admin ve lista, array vacío, user recibe 403, sin auth recibe 403)
